## 一、核心理念

### 为什么用网页而不是 AI 视频生成模型？

| 方案 | 优势 | 劣势 |
|------|------|------|
| **AI 视频生成模型** | 一键生成，视觉效果强 | 不可控（抽卡）、成本高、无法精确控制每帧 |
| **Notebook LM** | 简单易用 | 只能做静态图，无法做动画演示 |
| **Manim 等框架** | 代码驱动 | 限制模型发挥，不如直接写自由 |
| **网页（本方案）** | 字体/配色/节奏/每帧完全可控 | 需要开发能力 |

**核心结论**：网页 = 最可控的视频载体。改几行代码就能精确控制画面，比视频模型抽卡稳定得多，成本更低。

### 关键设计思路

1. **网页看起来像视频**：无页眉页码、无品牌标识、干净画面、隐藏式进度条
2. **文章 → 口播稿 → 开发计划 → 视觉演示 → 音频合成 → 录屏**
3. **口播稿定节奏，原文定密度**：双源原则
4. **每步视觉演示必须跟随口播稿节奏**：画面和声音天然对齐

## 二、Harness 六要素拆解

> Harness（驾驭工程）= 把模型能力、工具能力、人的判断组织成稳定可控的生产流程

### 1. 执行与编排 — 让模型知道下一步该做什么

**四阶段流水线 + 两个人工检查点**：

```
阶段一：内容编写
├─ 产出：口播稿 (script.md) + 开发计划 (outline.md)
├─ 口播稿：决定信息描述方式和整体节奏
└─ 开发计划：决定每章做几步，每步屏幕放什么
    ↓
[检查点 1] ← 强制停等
├─ 稿子和计划要不要改？
├─ 用哪套主题？
├─ 素材怎么准备？
└─ 后续开发模式？（逐章确认 / 顺序 / 并行）
    ↓
阶段二：开发
├─ 第一章必须先做 + 人工验收（基准锚点）
├─ 后续章节：逐章确认 / 顺序 / 并行 subagent
├─ 每章完成后逐项对照检查清单自检
└─ 全部过关才能交付
    ↓
[检查点 2] ← 是否需要自动合成音频？
    ↓
阶段三：音频合成（可选）
├─ 从口播稿对照网页步骤提取音频合成清单
├─ 调用 TTS 合成音频
└─ 每步一个音频文件，文件名与步骤编号一一对应
    ↓
阶段四：录屏
├─ 浏览器打开网页 + 开启音频自动播放
└─ 录屏产出音频和画面天然对齐的成品视频
```

### 2. 上下文管理 — 让模型看到最需要的信息

**分阶段加载，不一次性灌入所有文档**：

| 阶段 | 加载内容 | 不加载 |
|------|---------|--------|
| 阶段一 | 口播稿规范 + 原文 | 开发指引、音频规范 |
| 阶段二 | 开发指引 + 主题 + 当前章节大纲 | 口播稿规范、音频规范 |
| 阶段三 | 音频合成规范 | 开发指引 |
| 阶段四 | 录屏规范 | 其他所有 |

**效果**：模型在任何时候面前摆的都是最需要的信息，不被无关内容干扰。

### 3. 状态与记忆 — 跨步骤保持连续性

**三份文档各有分工**：

| 文档 | 作用 | 谁来写 | 谁来读 |
|------|------|--------|--------|
| `outline.md` | 跨步骤记忆载体，固定关键决定 | 阶段一产出 | 阶段二每章开发时读取 |
| `script.md` | 定节奏（先讲什么后讲什么） | 阶段一产出 | 阶段二、三读取 |
| `article.md` | 定信息密度（数据、案例从原文补） | 用户输入 | 阶段二读取 |

**文件化的工作记忆**：把关键状态写进文件，需要时再读回来。

### 4. 工具系统 — 让模型知道能做什么

**核心工具**：文件读写（agent 自带）

**多章节并行的隔离机制**：

| 隔离层 | 措施 |
|--------|------|
| 物理隔离 | 每章独立文件夹 |
| 样式隔离 | 每章独立 CSS 前缀，不抢类名 |
| 视觉统一 | 主题 token 兜底（字体/颜色/间距走全局变量） |

**效果**：各章节独立开发，互不影响，但视觉风格一致。

### 5. 约束与恢复 — 出错了怎么办

**反馈修复的最小切片原则**：

> 不要重做整章，先定位问题在哪一层，只改最小的切片。

| 问题层级 | 修复范围 |
|---------|---------|
| 节奏问题 | 只改口播稿对应段落的步调 |
| 视觉内容 | 只改某一步的画面 |
| 代码问题 | 只改对应组件 |

**反模式**：用户说"第 3 步节奏太快"→ 不要重做整个章节（会把做对的部分也改掉）。

### 6. 评估与观测 — 怎么知道自己做得对

**自评失真问题**：让写代码的 agent 自己评价，结果大概率是"还不错"。

**解决方案**：硬性自检规则

| 方案 | 实现方式 | 优先级 |
|------|---------|--------|
| **最优** | 开独立 reviewer agent，给产出文件 + 检查清单，从零逐项检查 | 推荐 |
| **次优** | 用 subagent 做质检 | 可接受 |
| **兜底** | 当前 agent 自己逐项核查（必须严格逐项，不允许目测放行） | 最低要求 |

## 三、工具链配置

### 四个核心工具

| 工具 | 用途 | 备注 |
|------|------|------|
| **AI Agent** | 核心执行 agent | 任何支持 skill 的 agent harness 均可 |
| **MiniMax Token Plan** | 国产模型替代方案 | Plus 极速版，速度快，量大管够 |
| **CC Switch** | 桌面配置工具，切换 agent 为任意模型 | 支持 AI Agent/Codex/Gemini CLI |
| **MMX CLI** | MiniMax 官方 CLI，语音合成 | 每天 9000 字额度，做两篇文章够用 |

### 多 Agent 能力

| 能力 | 特点 | 适用场景 | Token 开销 |
|------|------|---------|-----------|
| **Sub Agent** | 主 agent 分任务，子 agent 独立完成后交回结果，互相不可见 | 结果导向任务（并行开发多章节、一次性检查） | 低 |
| **Agent Teams** | 组长 + 组员，独立会话，可互相发消息共享任务列表 | 需要来回反馈的任务（质检：测试→开发→复查） | 高 |

**类比**：
- Sub Agent = 三个人分别去三个城市办事，办完回来汇报
- Agent Teams = 三个人拉进同一间会议室，边讨论边改方案

### 实战经验

| 经验 | 说明 |
|------|------|
| 并行 agent 数量 | 最多同时 3 个比较合适 |
| 模型选择 | 多 agent 调度需要能力较强的模型 |
| 音频合成 | 遇到速率限制会自动重试，已生成文件自动跳过，中途中断不用重来 |
| 第一章验收 | 必须亲自在浏览器从头点到尾，确认画面/节奏/信息密度/AI味 |
| 录屏模式 | 手动模式（自己录声音）/ 音频模式（audio=1）/ 自动模式（auto=1） |

## 四、Skill 设计模式总结

### 核心设计模式

| 模式 | 来源 | 说明 |
|------|------|------|
| **四阶段 + 两检查点** | 本教程 | 流水线设计，人工卡点 |
| **分阶段上下文加载** | 本教程 | 不一次性灌入所有文档 |
| **三文档分工** | 本教程 | outline 定结构、script 定节奏、article 定密度 |
| **物理隔离 + CSS 前缀** | 本教程 | 多章节并行不冲突 |
| **主题 token 兜底** | 本教程 | 全局变量保证视觉一致 |
| **最小切片修复** | 本教程 | 不重做整章，只改问题层 |
| **硬性自检 + 独立 reviewer** | 本教程 | 对抗自评失真 |
| **第一章强制验收** | 本教程 | 质量锚点，后续章节基准 |

### Harness 本质

> 做 harness 不是非得从零做一个 agent。能用一个 skill 约束 agent 把一个垂直工作做好，也是在做 harness。
